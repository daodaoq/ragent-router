# RAgent Router 项目改进建议报告

> 基于对 backend（Go）与 frontend（Electron + React）全量代码的审查生成。
> 按优先级 P0 → P3 排序，每项给出具体位置与改进方向。

---

## 一、整体评价

**亮点**：
- 架构清晰：`resilience/routing/proxy/store` 四层职责分明，韧性引擎（熔断/限流/重试/舱壁/超时）设计扎实，令牌桶 28ns/op 的热路径性能优秀。
- 依赖精简：后端仅 `modernc.org/sqlite` 一个直接依赖，纯 Go 无 CGO；前端技术栈选型现代（React 18 + Vite 5 + TS strict + Zustand）。
- 注释充分：后端每个包都有 package doc、算法复杂度说明、ASCII 流程图，是项目的一大亮点。
- 主题方案方向正确：CSS 变量 + light/dark 双主题。

**核心短板**：
1. **安全几乎裸奔**——无认证、CORS 全开、密钥管理薄弱。
2. **工程化基础设施缺失**——无配置文件、无结构化日志、无 metrics/tracing、无 CI/CD、前端无 ESLint/Prettier/测试。
3. **存在功能性 Bug**——请求日志 Prompt 永远为空、`proxyErr` 变量声明未赋值、`activate` 接口是空操作、监控接口返回硬编码 0。
4. **前端 API 层碎片化**——有统一封装却不用，16 处硬编码 URL，9 处静默吞错。
5. **i18n 名存实亡**——引入了完整框架却用 `lang === "zh" ?` 硬编码 111 处。
6. **测试覆盖严重不均**——后端 proxy/store/HTTP handler 零测试，前端完全无测试。

## 二、P0 — 紧急修复（安全与功能性 Bug）

### 2.1 [安全] 后端完全没有认证授权

**位置**：`backend/cmd/server/main.go` 全部 HTTP 路由

**问题**：所有 API 端点公开，任何人可查看请求日志、Token 用量、成本数据，CRUD 意图树，甚至直接用代理转发请求消耗你的 API 额度。

**建议**：
- 添加认证中间件，至少支持 API Token / Bearer Token 校验。
- Dashboard/监控/管理类接口必须鉴权。
- 代理接口 `/v1/messages` 可通过独立的 proxy token 控制。

### 2.2 [安全] CORS 配置过于宽松

**位置**：`backend/cmd/server/main.go:698-710`

```go
w.Header().Set("Access-Control-Allow-Origin", "*")
```

**问题**：任何网站都可跨域调用此 API，配合无认证可被任意网页盗用。

**建议**：限制为已知前端来源（Electron 的 `file://` 或本地 `http://localhost:5173`），通过配置项管理白名单。

### 2.3 [Bug] 请求日志 Prompt 永远为空字符串

**位置**：`backend/cmd/server/main.go:160`

```go
Prompt: store.CompactPrompt("", 500),  // 传入空字符串而非实际 prompt
```

**问题**：Dashboard 中所有请求日志都看不到用户问了什么，监控功能大打折扣。

**建议**：传入从请求体提取的 `prompt` 变量。

### 2.4 [Bug] `proxyErr` 变量声明但从未赋值

**位置**：`backend/internal/proxy/handler.go:248-266`

```go
var proxyErr error  // 永远为 nil
err = breaker.Call(func() error { ... })
if proxyErr != nil || err != nil {  // proxyErr 判断永远不成立
```

**问题**：错误日志只记录 `err`，可能遗漏熔断器打开等关键错误信息。

**建议**：删除冗余的 `proxyErr`，或正确赋值以区分熔断错误与上游错误。

### 2.5 [Bug] `/api/proxy/activate/{id}` 是空操作

**位置**：`backend/cmd/server/main.go:344-358`

**问题**：接口返回"已切换"，但实际什么都没做。Proxy 使用路由引擎动态选择供应商，没有"全局激活"概念。

**建议**：要么删除此接口，要么实现真正的切换逻辑（如临时锁定到某个供应商用于调试）。

### 2.6 [Bug] `/api/monitor/overview` 返回硬编码 0

**位置**：`backend/cmd/server/main.go:277-285`

```go
"today_requests": 0, "error_count": 0, "total_tokens": 0, "avg_latency_ms": 0
```

**问题**：监控页显示的全是 0，功能失效。

**建议**：从 SQLite 查询真实聚合数据。

### 2.7 [前端] 统一 API 层，消除 16 处硬编码 URL

**位置**：`frontend/src/api/index.ts` + 8 个组件文件

**问题**：已有 `request<T>` 封装却只有 dashboard/proxy 两个模块使用，其他组件直接 `fetch("http://localhost:15722/...")`，端口写死 16 处，无法通过环境变量配置。`vite.config.ts` 的 proxy 配置形同虚设。

**建议**：
- 所有 fetch 改用 `api/index.ts` 的 `request`。
- `BASE_URL` 改为 `import.meta.env.VITE_API_BASE`。
- 补全 `traffic/monitor/intent/setup/ccswitch` 五个 API 模块封装。

### 2.8 [前端] 引入 ErrorBoundary

**问题**：完全没有 ErrorBoundary，任何子组件抛错导致整个应用白屏。

**建议**：在 `App.tsx` 外层包裹 ErrorBoundary，给出友好的错误恢复 UI。

### 2.9 [前端] 清理死代码

**位置**：`frontend/src/pages/Dashboard.tsx` 及其依赖链

**问题**：Dashboard 未被 `App.tsx` 引用，连带 `CostOverview/CostTrend/ModelDistribution/RecentRoutes` 4 个组件、`stores/dashboard.ts`、`api/index.ts` 的 `dashboardApi/proxyApi` 都是死代码（约 6 个文件）。

**建议**：要么接入 Sidebar 使用，要么删除。

## 三、P1 — 高优先级（工程质量与可维护性）

### 3.1 [后端] 引入结构化日志

**问题**：全部用 `log.Printf`，无级别、无结构化字段，不利日志收集。

**建议**：Go 1.21+ 的标准库 `log/slog` 即可，输出 JSON，带 level/trace_id 字段。

### 3.2 [后端] 引入配置文件机制

**位置**：`backend/cmd/server/main.go` 多处硬编码

**问题**：
- 韧性参数硬编码（限流 100 QPS、熔断阈值 0.5、重试 2 次、连接池 100/20）。
- 成本费率硬编码在 `proxy/handler.go:453-461`。
- 默认供应商配置硬编码。

**建议**：引入 `viper` 或简单的 YAML 解析，支持 `config.yaml` + 环境变量覆盖。

### 3.3 [后端] 修复 goroutine 泄漏

**位置**：`backend/internal/routing/embedding.go:203-209`

```go
func (c *EmbeddingCache) evictLoop(interval time.Duration) {
    for range ticker.C {  // 无退出机制
```

**问题**：`ReloadIntents` 创建新 cache 时，旧 cache 的 goroutine 永不停止。

**建议**：接收 context，`select { case <-ctx.Done(): return; case <-ticker.C: ... }`。

### 3.4 [后端] 补充核心模块单元测试

**缺失**：`proxy/`、`store/`、`cmd/server/main.go` 的 HTTP handler 完全无测试。

**建议**：
- proxy 层：测试适配器翻译、SSE 流式转发、Token 解析。
- store 层：用内存 SQLite 测试 CRUD 和查询。
- handler 层：用 `httptest` 测试关键 API。

### 3.5 [前端] i18n 全面修正

**位置**：`IntentPanel.tsx`(42处)、`Settings.tsx`(36处)、`MonitorPanel.tsx`(20处) 等

**问题**：111 处 `lang === "zh" ? "中文" : "English"` 硬编码，完全绕开了已引入的 i18next 框架。

**建议**：将所有硬编码字符串迁入 `locales/zh.json` 和 `locales/en.json`，统一用 `t()` 调用。

### 3.6 [前端] 引入 ESLint + Prettier + husky

**问题**：无任何代码规范工具，代码质量完全依赖个人自觉。

**建议**：
- ESLint + `@typescript-eslint` + React 插件。
- Prettier 统一格式。
- husky + lint-staged 在 commit 时自动检查。

### 3.7 [前端] 为 `window.electronAPI` 声明全局类型

**位置**：4 处 `@ts-ignore`（`stores/app.ts:20`、`Settings.tsx:18`、`StatusBar.tsx:4`、`TitleBar.tsx:5`）

**建议**：新建 `src/global.d.ts`，声明 `ElectronAPI` 接口和 `window.electronAPI` 类型。

### 3.8 [前端] 引入路由库

**问题**：用 `useState` 手动切换页面，无 URL、无前进后退、无深链接、页面状态不保持。

**建议**：引入 `react-router-dom`，或至少用 URL hash 保持当前页。

### 3.9 [前端] 用 React.lazy 做代码分割

**位置**：`frontend/src/App.tsx:2-10`

**问题**：5 个页面同步 import，首次加载打包所有代码。

**建议**：`const Providers = React.lazy(() => import("./components/Providers"))` + `<Suspense>`。

### 3.10 [文档] 修正 README 与代码不一致

**位置**：`README.md` 声称后端是 Go，但 `electron/main.js:101` 实际 spawn 的是 Python uvicorn。

**建议**：核实实际后端实现，统一文档描述。

## 四、P2 — 中优先级（性能、可观测性、体验）

### 4.1 [后端] 添加 Prometheus metrics 导出

**问题**：只有自定义统计接口，无标准 `/metrics` 端点，无法接入 Prometheus/Grafana 体系。

**建议**：引入 `prometheus/client_golang`，暴露请求计数、延迟直方图、熔断器状态、令牌桶剩余等指标。

### 4.2 [后端] 添加分布式 tracing

**问题**：请求从客户端到代理到上游的调用链完全不可追踪。

**建议**：引入 OpenTelemetry，在请求入口生成 trace_id，透传到上游和日志。

### 4.3 [后端] 完善健康检查

**位置**：`backend/cmd/server/main.go` `/health` 端点

**问题**：只返回静态 `{"status":"ok"}`，不检查 DB 连接、供应商可达性。

**建议**：区分 liveness（/healthz）和 readiness（/readyz），后者检查依赖。

### 4.4 [后端] 按粒度限流

**问题**：只有全局 100 QPS 限流，无按用户/IP/供应商/Token 数的细粒度限流。Dashboard API 无限流保护。

**建议**：复用现成的分片令牌桶，按维度限流。

### 4.5 [后端] 修复 SSE 跨 chunk 解析缺陷

**位置**：`backend/internal/proxy/token_tracker.go:104`（注释自承认）

**问题**：不处理跨 TCP chunk 的 SSE 事件，可能导致 Token 用量统计不准。

**建议**：用 `bufio.Scanner` + 跨 chunk 缓冲区。

### 4.6 [后端] `writeJSON` 先编码再 WriteHeader

**位置**：`backend/cmd/server/main.go:770-776`

**问题**：先 WriteHeader 再 Encode，若编码失败客户端收到空 body + 200 状态码。

**建议**：先 encode 到 `bytes.Buffer`，确认无错后再 WriteHeader。

### 4.7 [后端] 请求 ID 用 UUID 而非 UnixNano

**位置**：`proxy/handler.go:238`、`cmd/server/main.go:159`

**问题**：`time.Now().UnixNano()` 高并发下可能碰撞。go.mod 间接依赖已有 `google/uuid` 但未使用。

**建议**：改用 `uuid.NewString()`。

### 4.8 [前端] 拆分巨型组件

| 组件 | 行数 | 建议 |
|---|---|---|
| `IntentPanel.tsx` | 599 | 拆出 `NodeEditModal`，分离树构建/分类/CRUD 逻辑 |
| `Settings.tsx` | 482 | 按 5 个区块拆为 `SystemStatusCard/CCSwitchCard/AppearanceCard/BackendCard/DataLogCard` |
| `TrafficMonitor.tsx` | 355 | 拆为 3 个区块组件 + 1 个图表组件 |

### 4.9 [前端] 引入 react-query 管理服务端状态

**问题**：5 个页面全部用本地 `useState` 管理服务端数据，无缓存、无跨页面共享、手动管理 loading/error/data 三态。

**建议**：引入 `@tanstack/react-query`，统一数据获取、缓存、失效、重试。

### 4.10 [前端] 修正错误处理

**位置**：9 处 `.catch(() => {})` 静默吞错

**建议**：移除空 catch，加全局错误提示（message.error），提供重试按钮。

### 4.11 [前端] 环境变量管理

**问题**：无 `.env` 文件，端口 15722 硬编码在 8 个文件 + `electron/main.js` + `vite.config.ts`。

**建议**：新建 `.env.development`/`.env.production`，用 `VITE_API_PORT`/`VITE_API_BASE`。

### 4.12 [前端] 补充测试

**建议**：引入 `vitest` + `@testing-library/react`，先覆盖 store 和关键组件（Providers、IntentPanel 的 CRUD）。

## 五、P3 — 低优先级（代码整洁与细节）

### 5.1 [后端] 消除重复造轮子

- `proxy/handler.go:472-483` `contains`/`searchString` → 用 `strings.Contains`。
- `proxy/handler.go:429-437` `joinStrings` → 用 `strings.Join`。

### 5.2 [后端] 添加 `.gitignore`

**问题**：`backend/server.exe`（15MB 编译产物）被提交到仓库。

### 5.3 [后端] 添加 Makefile / Dockerfile

**建议**：支持 `make build`、`make test`、`docker build`。

### 5.4 [后端] `RuleEngine.AddRule/RemoveRule` 加锁

**位置**：`internal/routing/matcher.go:172-186`

**问题**：并发修改 `e.rules` 切片有数据竞争。

### 5.5 [后端] 成本节省估算逻辑化

**位置**：`internal/store/sqlite.go:130-132`

```go
overview.SavedAmount = overview.MonthCost * 0.25  // 固定 25%
```

**建议**：基于"如果全部走 Claude 的费用"vs"实际混合路由后的费用"真实计算。

### 5.6 [前端] 样式重构

**问题**：
- 内联 `style={{}}` 泛滥。
- `index.css` 用 `!important` 覆盖 Ant Design 20+ 处，脆弱。
- 主题变量在 `index.css` 和 `main.tsx` 重复定义。

**建议**：抽公共 CSS 类（`.card-base` 等），减少内联样式；考虑 CSS Modules 或 Tailwind。

### 5.7 [前端] 可访问性 (a11y)

**问题**：无 `aria-label`、`aria-current`，`<html lang="en">` 与默认中文不符，`PageHelp` 的 `?` 按钮是 `<span>` 不可聚焦。

### 5.8 [前端] `electron/main.js` 改为 TypeScript

**问题**：项目声称用 TS 但主进程是 `.js`，`tsconfig.json` 的 `include: ["electron/**/*.ts"]` 实际无效。

### 5.9 [前端] CSP 收紧

**位置**：`index.html:7-8`

**问题**：`script-src 'self' 'unsafe-inline'` 允许内联脚本。

### 5.10 [前端] 轮询优化

**位置**：`MonitorPanel.tsx:68-71` 10s 轮询不受开关控制，页面不可见时仍持续请求。

**建议**：用 `document.visibilitychange` 事件暂停轮询，或用 `react-query` 的 `refetchInterval` + `refetchOnWindowFocus`。

### 5.11 [前端] 移除 console.log 残留

**位置**：`stores/dashboard.ts:48`、`IntentPanel.tsx:73`、`MonitorPanel.tsx:60`

### 5.12 [前端] 统一 provider 颜色映射

**位置**：`MonitorPanel.tsx:81` 大小写不一致（`DeepSeek` vs `deepseek`），且在文件内重复定义两次。

**建议**：抽为 `constants/providerColors.ts` 统一管理。

### 5.13 [CI/CD] 添加 GitHub Actions

**建议**：lint + typecheck + test + build 流水线，PR 必须通过。
