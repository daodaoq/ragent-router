### 1.1【P0】关键词路由规则 Provider 大小写不匹配 —— 路由层完全失效

**位置**：`internal/routing/matcher.go:211,217,223,229,237,243`

**问题**：`DefaultRules()` 中所有规则的 `Provider` 字段用的是小写：

```go
// matcher.go:211
Provider: "claude",   // 小写
// matcher.go:237
Provider: "deepseek", // 小写
```

但供应商注册表的 key 来自 `main.go:64`：

```go
providerMap[providers[i].Name] = &providers[i]  // Name = "Claude" / "DeepSeek"（首字母大写）
```

而 `matcher.go:124` 的查找是**大小写敏感**的精确匹配：

```go
if prov, ok := e.providers[rule.Provider]; ok && prov.Enabled {
```

`e.providers["claude"]` 永远找不到 `"Claude"`。**后果**：所有关键词命中的请求（"架构"、"debug"、"生成"等）都无法找到供应商，穿透到 Embedding/LLM 层或默认回退。如果没配 Embedding 和 Classifier，全部走 DeepSeek——"架构设计该用 Claude"这个核心卖点完全失效。

**验证方式**：启动服务后发一个包含 "architecture" 的请求，看日志中 `[路由]` 是否打到了 Claude。

**修复**：统一用大写，或在 `NewRuleEngine` 中做大小写归一化：

```go
// 方案 A：修改 DefaultRules（推荐）
Provider: "Claude",    // 与供应商 Name 一致
Provider: "DeepSeek",

// 方案 B：在 NewRuleEngine 中归一化（更健壮）
// 建立小写 name → 原始 name 的映射，匹配时用小写查找
```

### 1.2【P0】OpenAI/DeepSeek 的 SSE Token 统计失败 —— 成本统计为 0

**位置**：`internal/proxy/token_tracker.go:147-177, 186-212`

**问题**：`processEvent` 只处理 Anthropic 的事件类型（`message_start`/`message_delta`/`message_stop`），而 `extractUsage` 只读取 Anthropic 的字段名：

```go
// token_tracker.go:197-208
if v, ok := usage["input_tokens"].(float64); ok { ... }   // Anthropic 字段名
if v, ok := usage["output_tokens"].(float64); ok { ... }  // Anthropic 字段名
```

但 OpenAI/DeepSeek 的 SSE 流使用不同的事件类型和字段名：
- 事件类型是 `chat.completion.chunk`，不是 `message_delta`
- usage 字段是 `prompt_tokens` / `completion_tokens`，不是 `input_tokens` / `output_tokens`
- usage 只在最后一个 chunk 中出现（需要 `stream_options: {"include_usage": true}`）

**后果**：经 DeepSeek/OpenAI 供应商的请求，`tracking.Usage` 全部为 0，`estimateCost` 返回 0，Dashboard 中的费用和 Token 统计全部缺失。

**修复**：在 `extractUsage` 中同时兼容两种字段名：

```go
func (t *TokenTracker) extractUsage(payload map[string]interface{}) {
    raw, _ := payload["usage"].(map[string]interface{})
    if raw == nil { return }

    // 兼容 Anthropic (input_tokens) 和 OpenAI (prompt_tokens)
    getInput := func() int {
        for _, k := range []string{"input_tokens", "prompt_tokens"} {
            if v, ok := raw[k].(float64); ok { return int(v) }
        }
        return 0
    }
    // ... 同理处理 output
}
```

同时 `processEvent` 的 `default` 分支已尝试提取，但需要确认 OpenAI 的事件能走到这里。

### 1.3【P0】SSE 流式传输中重试导致响应混乱

**位置**：`internal/proxy/handler.go:248-253, 359-374`

**问题**：`doUpstreamRequest` 在 Line 359 调用了 `w.WriteHeader(http.StatusOK)` 开始 SSE 流式传输，之后 Line 368 `io.Copy` 如果失败会返回 error。这个 error 会被 `retry.Do`（Line 250）捕获并触发重试：

```go
// handler.go:249-253
err = breaker.Call(func() error {
    return retry.Do(r.Context(), backoff, backoff.MaxAttempts(), func() error {
        return p.doUpstreamRequest(w, r, provider, bodyJSON, tracking)  // 已写 200 头
    })
})
```

重试时会再次调用 `doUpstreamRequest`，再次 `w.WriteHeader(200)` → `http: superfluous response.WriteHeader` 报错，且客户端会收到混乱的 SSE 数据（两次响应拼接）。

**后果**：上游网络抖动导致 `io.Copy` 中途失败时，客户端收到损坏的 SSE 流，可能引发解析错误。

**修复**：SSE 流式传输一旦开始（WriteHeader 后）就不能重试。应该区分"请求建立阶段"和"流式传输阶段"的错误：

```go
// 只在 WriteHeader 之前的错误才重试
func (p *Proxy) doUpstreamRequest(...) error {
    resp, err := p.client.Do(upstreamReq)
    if err != nil { return err }  // 可重试

    if resp.StatusCode >= 400 {
        return fmt.Errorf("upstream HTTP %d", resp.StatusCode)  // 可重试
    }

    // ── 不可重试区：WriteHeader 后的错误直接返回特殊标记 ──
    w.WriteHeader(http.StatusOK)
    _, err = io.Copy(tracker, resp.Body)
    if err != nil {
        return ErrStreamStarted  // 标记：流已开始，不可重试
    }
    return nil
}

// retry.Do 中判断
retry.Do(..., func() error {
    err := p.doUpstreamRequest(...)
    if errors.Is(err, ErrStreamStarted) {
        return retry.Break(err)  // 终止重试
    }
    return err
})
```

### 2.1【P0】Match 读 `r.intents` 无锁 —— 数据竞争

**位置**：`internal/routing/hybrid_engine.go:276`


### 2.2【P0】EmbeddingCache goroutine 泄漏

**位置**：`internal/routing/embedding.go:143, 203-209`

```go
func NewEmbeddingCache(ttl time.Duration) *EmbeddingCache {
    c := &EmbeddingCache{...}
    go c.evictLoop(10 * time.Minute)  // 启动后台 goroutine
    return c
}

func (c *EmbeddingCache) evictLoop(interval time.Duration) {
    ticker := time.NewTicker(interval)
    defer ticker.Stop()
    for range ticker.C {  // 无退出机制
        c.evictExpired()
    }
}
```

`hybrid_engine.go:233` 的 `ReloadIntents` 每次创建新 cache：

```go
r.cache = NewEmbeddingCache(1 * time.Hour)  // 旧 cache 的 goroutine 永不停止
```

用户每次通过 Dashboard 修改意图树，就泄漏一个 goroutine。

**修复**：给 `EmbeddingCache` 加 `context.Context` 或 `Stop()` 方法：

```go
type EmbeddingCache struct {
    mu      sync.RWMutex
    entries map[string]cacheEntry
    ttl     time.Duration
    stop    chan struct{}
}

func NewEmbeddingCache(ttl time.Duration) *EmbeddingCache {
    c := &EmbeddingCache{
        entries: make(map[string]cacheEntry),
        ttl:     ttl,
        stop:    make(chan struct{}),
    }
    go c.evictLoop(10 * time.Minute)
    return c
}

func (c *EmbeddingCache) Stop() { close(c.stop) }

func (c *EmbeddingCache) evictLoop(interval time.Duration) {
    ticker := time.NewTicker(interval)
    defer ticker.Stop()
    for {
        select {
        case <-c.stop:
            return
        case <-ticker.C:
            c.evictExpired()
        }
    }
}

// ReloadIntents 中：
oldCache := r.cache
r.cache = NewEmbeddingCache(...)
oldCache.Stop()  // 停止旧 goroutine
```

### 3.3【P1】无 panic recovery 中间件

**修复**：

```go
func recoveryMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        defer func() {
            if err := recover(); err != nil {
                log.Printf("[PANIC] %v\n%s", err, debug.Stack())
                http.Error(w, `{"error":"internal server error"}`, 500)
            }
        }()
        next.ServeHTTP(w, r)
    })
}
```

```go
}
```

中文字符 3 字节，`prompt[:500]` 可能截断到字符中间，产生乱码。且两处截断长度不一致（500 vs 200）。

**修复**：用 `utf8.RuneCountInString` + 按 rune 截断：

```go
runes := []rune(prompt)
if len(runes) <= maxLen { return prompt }
return strings.TrimSpace(string(runes[:maxLen])) + "..."
```

## 五、可观测性缺失

### 5.1【P1】无结构化日志

全部用 `log.Printf`，无级别、无 JSON 结构。Go 1.21+ 的 `log/slog` 即可。

### 5.2【P1】关键路径缺少日志

- 限流拒绝（handler.go:168-170）无日志
- 舱壁拒绝（handler.go:181-183）无日志
- 上游 4xx/5xx（handler.go:348-351）无日志

### 5.3【P2】无 Prometheus metrics

只有自定义统计接口，无 `/metrics` 端点。

### 5.4【P2】健康检查过于简单

`/health` 只返回静态 `{"status":"ok"}`，不检查 DB 连接、供应商可达性。应区分 liveness（`/healthz`）和 readiness（`/readyz`）。

### 5.5【P2】无分布式 tracing

请求从客户端到代理到上游的调用链不可追踪。建议引入 OpenTelemetry。

## 九、亮点（值得保持）

- **韧性引擎设计扎实**：令牌桶 28ns/op 热路径、三态熔断 + 滑动窗口、级联超时、3 种 Jitter 策略，工程质量高。
- **注释极其充分**：每个包有 package doc，关键算法有数学公式和复杂度分析，ASCII 流程图清晰。
- **依赖精简**：仅 `modernc.org/sqlite` 一个直接依赖，纯 Go 无 CGO。
- **适配器模式**：`ProviderAdapter` 接口抽象了协议差异，扩展性好。
- **分片并发存储**：FNV-64a 哈希 + 2048 分片 + Double-Checked Locking，并发设计考究。

问题主要集中在**协议兼容性**（OpenAI 格式）、**并发边界**（热重载时的数据竞争）、**工程化基础设施**（安全/日志/配置/测试）三方面。核心韧性引擎本身质量很高。
