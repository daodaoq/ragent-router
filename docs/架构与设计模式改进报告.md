# RAgent Router 架构与设计模式改进报告

> 审查时间：2026-07-19
> 审查范围：`backend/` 全部 20 个 Go 源文件
> 审查视角：**长期演进、架构健康度、设计模式应用机会**（非致命 Bug 审查，假定上一轮 P0 已修复）

---

## 0. 总体评价

代码质量相比首轮审查有明显提升：新增了 `bulkhead` 舱壁、`LLMIntentClassifier`、`intents.go` 类型层、`hybrid_engine_test.go`，韧性组件补齐到五件套（限流/舱壁/熔断/重试/超时），注释密度和算法复杂度分析在 Go 项目中属于上乘。

但从**长期演进**视角看，当前结构存在三个核心问题：

1. **`main.go` 过度膨胀（944 行）**，承担了 HTTP 路由、中间件、配置加载、DTO 转换、业务辅助函数等 5+ 种职责，违反单一职责原则。
2. **包间存在反向依赖**：`routing` 包 import 了 `proxy` 包（`proxy.ProviderConfig`），而 `proxy` 又通过 `RouteMatcher` 接口调用 `routing`，形成隐性的双向耦合。
3. **扩展点未抽象**：新增路由策略、新增供应商、切换存储后端这三类最常见的演进需求，当前都需要改核心代码，不符合开闭原则。

下面按维度展开。

## 1. 架构分层问题

### 1.1 `main.go` 是一个"上帝文件"

**现状**：`cmd/server/main.go` 共 944 行，承担以下职责：

| 行号区间 | 职责 | 应归属的包 |
|---|---|---|
| 41-156 | 启动流程、配置加载、组件装配 | `main`（合理） |
| 158-179 | 请求日志回调（DTO 转换 + 存储） | `internal/api` 或 `internal/logsvc` |
| 181-511 | 20+ 个 HTTP handler 内联定义 | `internal/api/handlers_*.go` |
| 518-654 | 意图树 CRUD handler | `internal/api/handlers_intent.go` |
| 656-697 | 服务器启动 + 优雅关闭 | `main`（合理） |
| 703-772 | 供应商配置加载 | `internal/config` |
| 778-866 | 中间件定义（4 个） | `internal/api/middleware.go` |
| 872-925 | 意图记录转换 + 热重载工具 | `internal/routing` 或 `internal/intentsvc` |
| 929-943 | `writeJSON` 工具函数 | `internal/api/respond.go` |

**问题**：
- HTTP handler 无法单元测试（内联在 main 里，无法导入）
- 中间件无法复用到其他 server
- 配置加载逻辑和启动逻辑混在一起
- `writeJSON`、`reloadIntents` 这种工具函数被锁死在 main 包

**建议结构**：

```
cmd/server/main.go           ← 只做装配，<200 行
internal/config/config.go    ← Config 结构体 + Load()
internal/api/
  router.go                  ← 路由注册
  middleware.go              ← recovery/requestID/auth/cors
  respond.go                 ← writeJSON, writeError
  handlers_proxy.go          ← /v1/messages, /api/proxy/*
  handlers_dashboard.go      ← /api/dashboard/*
  handlers_monitor.go        ← /api/monitor/*
  handlers_intent.go         ← /api/intent/*
  handlers_resilience.go     ← /api/resilience/*, /api/routing/*
internal/intentsvc/service.go ← 意图转换 + 热重载
```

### 1.2 包间反向依赖

**现状**：

```
cmd/server
  └── proxy          ← 代理核心
        ├── routing  ← 路由引擎（import proxy.ProviderConfig）
        ├── store    ← 持久化
        └── resilience
```

`proxy/handler.go:84` 定义了 `RouteMatcher` 接口，`routing/hybrid_engine.go` 实现它——这是正确的依赖倒置。但 `routing` 包内部又 import 了 `proxy`：

```go
// routing/hybrid_engine.go:41
import "github.com/ragent/router/internal/proxy"

// routing/hybrid_engine.go:70
providers map[string]*proxy.ProviderConfig

// routing/hybrid_engine.go:262
func (r *HybridRouter) Match(...) *proxy.ProviderConfig { ... }
```

这导致 `proxy` 和 `routing` 形成**双向耦合**：
- `proxy` 依赖 `routing` 的 `HybridRouter`（通过接口）
- `routing` 依赖 `proxy.ProviderConfig`（直接引用具体类型）

**风险**：
- 任何对 `proxy.ProviderConfig` 字段的修改都会触发 `routing` 重新编译
- 无法单独测试 `routing` 包（必须 stub `proxy.ProviderConfig`）
- 将来 `proxy` 包如果引入更多依赖，`routing` 会被动拖累

**建议**：把 `ProviderConfig` 提取到独立的 `internal/model` 或 `internal/provider` 包：

```
internal/provider/
  types.go    ← ProviderConfig, ProviderInfo 接口
  errors.go   ← 供应商相关错误
```

然后 `proxy` 和 `routing` 都依赖 `provider` 包，两者互不依赖：

```
proxy    → provider
routing  → provider
```

## 2. 设计模式应用机会

### 2.1 策略模式（部分应用，可深化）

**当前**：`RouteMatcher` 接口是策略模式的体现，但 `HybridRouter` 内部的三个策略（关键词/Embedding/LLM）是硬编码的 if 链：

```go
// routing/hybrid_engine.go:262-304
func (r *HybridRouter) Match(...) *proxy.ProviderConfig {
    if prov := r.keywordEngine.Match(...); prov != nil { ... }   // 策略1
    if hasEmbeddingSvc && hasIntents { ... }                      // 策略2
    if r.classifier != nil { ... }                                // 策略3
    return r.fallback()                                           // 兜底
}
```

**问题**：新增一个策略（比如"基于历史的路由"、"基于成本的动态路由"、"基于模型负载的路由"）需要改 `Match` 方法体，违反开闭原则。

**建议**：抽象出统一的策略接口，HybridRouter 持有策略链：

```go
// routing/strategy.go
type RouteStrategy interface {
    Name() string
    Match(ctx context.Context, prompt string, model string) (*provider.Config, MatchResult)
}

type MatchResult struct {
    Matched  bool
    Provider *provider.Config
    Reason   string
    Latency  time.Duration
}

// HybridRouter 持有有序策略链
type HybridRouter struct {
    strategies []RouteStrategy  // 按顺序执行，首个 Matched=true 即返回
    fallback   FallbackStrategy
}

func (r *HybridRouter) Match(ctx context.Context, prompt, model string) *provider.Config {
    for _, s := range r.strategies {
        result := s.Match(ctx, prompt, model)
        if result.Matched {
            atomic.AddInt64(&r.hits[s.Name()], 1)
            return result.Provider
        }
    }
    return r.fallback.Match(ctx, prompt, model)
}
```

**收益**：
- 新增策略 = 新增一个 struct + 注册到 strategies 切片，零改动核心代码
- 策略可独立测试
- 策略顺序可配置（通过 config 文件调整）
- 统计自动按策略名分桶

### 2.2 责任链模式（韧性层可重构）

**当前**：韧性组件是嵌套调用，本质是责任链，但实现是硬编码的：

```go
// proxy/handler.go:179-196
if !p.limiter.Allow() { return }              // 1. 限流
err := p.bulkhead.Execute(ctx, func() error { // 2. 舱壁
    return p.handleRequest(w, r)              //    内部还有：
    //   3. breaker.Call(doUpstreamRequest)
    //      4. retry.Do(timeout.Cascading(...))
})
```

**问题**：
- 扩展性差：新增一个韧性层（比如 quota、admission control）要改 `ServeHTTP` 和 `handleRequest` 两处
- 顺序固定：无法通过配置调整执行顺序
- 难以单独测试某个韧性层

**建议**：抽象成中间件链（和 HTTP middleware 一致的思想）：

```go
// proxy/pipeline.go
type ProxyHandler func(ctx context.Context, req *Request) (*Response, error)
type Middleware func(next ProxyHandler) ProxyHandler

func RateLimit(limiter *ratelimit.TokenBucket) Middleware { ... }
func Bulkhead(b *bulkhead.Bulkhead) Middleware { ... }
func CircuitBreaker(cb *circuitbreaker.CircuitBreaker) Middleware { ... }
func Retry(cfg retry.Config) Middleware { ... }
func Timeout(d time.Duration) Middleware { ... }

// 装配
pipeline := Chain(
    RateLimit(p.limiter),
    Bulkhead(p.bulkhead),
    CircuitBreaker(p.breaker),
    Retry(retryCfg),
    Timeout(120*time.Second),
)
pipeline(p.forward)
```

**收益**：韧性层可任意组合、顺序可调、每个中间件可独立测试。

### 2.3 观察者模式（日志回调可扩展）

**当前**：`Proxy.OnRequestLog` 是单回调函数：

```go
// proxy/handler.go:74
OnRequestLog func(log RequestLog)
```

**问题**：只能挂一个观察者。将来要同时做"写日志 + 发 Prometheus 指标 + 触发告警 + 更新成本看板"，要么在回调里塞所有逻辑，要么用全局变量。

**建议**：

```go
// proxy/observer.go
type RequestObserver interface {
    OnComplete(ctx context.Context, log RequestLog)
}

type ObserverGroup struct {
    observers []RequestObserver
}

func (g *ObserverGroup) OnComplete(ctx context.Context, log RequestLog) {
    for _, o := range g.observers {
        o.OnComplete(ctx, log)  // 异步或同步，可配置
    }
}

// 实现示例
type LogStoreObserver struct{ store *store.LogStore }
type MetricsObserver struct{ prom prometheus.Counter }
type AlertObserver struct{ alerter Alerter }
```

### 2.4 仓储模式（分离持久化与业务查询）

**当前**：`LogStore` 混合了 CRUD 和业务查询：

```go
// store/sqlite.go
func (s *LogStore) Insert(log *RequestLogRecord) error        // CRUD
func (s *LogStore) DashboardOverview() (*DashboardOverview, error)  // 业务查询
func (s *LogStore) ModelDistribution() ([]ModelDistribution, error) // 业务查询
func (s *LogStore) CostTrend(days int) ([]CostTrendPoint, error)    // 业务查询
func (s *LogStore) MonitorOverview() (map[string]interface{}, error) // 业务查询
```

**问题**：
- 切换存储后端（PostgreSQL/MySQL）需要重写整个类
- 业务查询方法直接返回 DTO，和 HTTP 层耦合（改字段名要改两层）
- `MonitorOverview` 返回 `map[string]interface{}`，丢失类型安全

**建议**：分离接口

```go
// store/repository.go
type RequestLogRepository interface {
    Insert(ctx context.Context, log *RequestLogRecord) error
    FindByFilter(ctx context.Context, filter LogFilter) ([]*RequestLogRecord, error)
    Aggregate(ctx context.Context, spec AggregateSpec) (AggregateResult, error)
}

// internal/dashboard/service.go
type DashboardService struct {
    repo store.RequestLogRepository
}

func (s *DashboardService) Overview(ctx context.Context) (*DashboardOverview, error) {
    // 调用 repo.Aggregate，组合业务逻辑
}
```

### 2.5 选项模式（Options Pattern）

**当前**：`NewProxy(cfg Config)` 用配置结构体，字段会越来越多：

```go
// proxy/handler.go:110
type Config struct {
    Providers             []ProviderConfig
    Matcher               RouteMatcher
    HTTPClient            *http.Client
    GlobalRateLimit       float64
    MaxConcurrentRequests int
    // 将来要加：Logger, Metrics, Tracer, CostTable, QuotaStore...
}
```

**建议**：改用 Functional Options：

```go
type Option func(*Proxy)

func WithRateLimit(rps float64) Option { ... }
func WithBulkhead(max int) Option { ... }
func WithLogger(l *slog.Logger) Option { ... }
func WithMetrics(m Metrics) Option { ... }

p := NewProxy(providers, matcher,
    WithRateLimit(100),
    WithBulkhead(50),
    WithLogger(logger),
)
```

**收益**：新增选项不破坏现有调用方，默认值清晰，可选参数显式。

### 2.6 工厂模式的小瑕疵

**当前**：`AdapterFactory` 每次调用 `Get` 都创建新适配器实例：

```go
// proxy/adapter.go:213
func (f *AdapterFactory) Get(providerName string) ProviderAdapter {
    if adapter, ok := f.adapters[providerName]; ok {
        return adapter
    }
    return AdapterFor(providerName)  // 每次返回 &AnthropicAdapter{} 或 &OpenAIAdapter{}
}
```

适配器本身**无状态**，每次分配新对象增加 GC 压力（虽然实际影响小）。

**建议**：用 sync.Map 缓存或预注册单例：

```go
var (
    anthropicAdapter = &AnthropicAdapter{}
    openaiAdapter    = &OpenAIAdapter{}
)

func AdapterFor(name string) ProviderAdapter {
    if isOpenAICompat(name) {
        return openaiAdapter
    }
    return anthropicAdapter
}
```

## 3. 可扩展性评估

### 3.1 新增供应商的扩展成本

**当前**：新增一个供应商（比如 Gemini）需要改 4 处分散的代码：

| 步骤 | 改动位置 | 风险 |
|---|---|---|
| 1. 环境变量加载 | `main.go:721-746` `loadProviders` | 容易遗漏 |
| 2. 适配器识别 | `adapter.go:188-197` `AdapterFor` 的关键词列表 | 容易遗漏 |
| 3. 成本费率 | `handler.go:486-491` `estimateCost` 的 rates map | 容易遗漏 |
| 4. 路由规则（可选） | `matcher.go:208-249` `DefaultRules` | 按需 |

**问题**：这 4 处分散在不同包里，没有任何机制保证一致性。新增 Gemini 时如果忘了改 `estimateCost`，费用统计就静默归零（这正是上一轮 DeepSeek 出过的问题）。

**建议**：把供应商的所有信息聚合成一个 `Provider` 抽象：

```go
// internal/provider/provider.go
type Provider struct {
    Config  Config          // ID, Name, BaseURL, APIKey, Model
    Adapter Adapter         // 协议适配器
    Cost    CostTable       // 输入/输出费率
    Rules   []routing.Rule  // 可选：默认路由规则
}

type Registry struct {
    providers map[string]*Provider
}

func (r *Registry) Register(p *Provider) error { ... }
func (r *Registry) Get(name string) (*Provider, bool) { ... }
```

新增供应商 = 注册一个 `Provider` 实例，所有相关信息集中一处。

### 3.2 新增路由策略的扩展成本

如 2.1 所述，当前需要改 `HybridRouter.Match` 方法体。采用策略链后，零改动核心代码。

### 3.3 切换存储后端的扩展成本

**当前**：`LogStore` 和 `IntentStore` 直接依赖 `*sql.DB` + 硬编码 SQL。切换 PostgreSQL 需要：
- 新建 `postgres.go`，重写所有方法
- 改 `main.go` 的初始化代码
- 无法在编译期保证两个实现的接口一致

**建议**：定义接口 + 多实现：

```go
// store/interfaces.go
type LogStore interface {
    Insert(ctx context.Context, log *RequestLogRecord) error
    DashboardOverview(ctx context.Context) (*DashboardOverview, error)
    // ...
}

// store/sqlite_log.go
type SQLiteLogStore struct{ db *sql.DB }
// implements LogStore

// store/postgres_log.go（将来）
type PostgresLogStore struct{ db *sql.DB }
// implements LogStore
```

### 3.4 配置热更新能力

**当前**：只有意图树支持热重载（`ReloadIntents`）。供应商、路由规则、韧性参数修改都需要重启服务。

**建议**：
- 供应商配置支持运行时增删（通过 API + `ProviderRegistry`）
- 韧性参数支持动态调整（熔断阈值、限流速率等）
- 采用 `fsnotify` 监听配置文件变化，自动热重载

## 4. 接口设计问题

### 4.1 `RouteMatcher` 返回具体类型

```go
// proxy/handler.go:83
type RouteMatcher interface {
    Match(ctx context.Context, prompt string, model string) *ProviderConfig
}
```

返回 `*ProviderConfig`（proxy 包的具体类型）导致 `routing` 必须依赖 `proxy`。应返回接口或独立类型（见 1.2）。

### 4.2 缺少统一的 Handler 类型

**当前**：所有 HTTP handler 是 `func(w http.ResponseWriter, r *http.Request)`，错误处理每个 handler 都要重复：

```go
// main.go:262-268（典型模板）
overview, err := logStore.DashboardOverview()
if err != nil {
    writeJSON(w, http.StatusInternalServerError, map[string]string{"error": err.Error()})
    return
}
writeJSON(w, http.StatusOK, overview)
```

这个模板在 main.go 里重复了 15+ 次。

**建议**：定义带错误返回的 Handler 类型 + 统一错误处理：

```go
// internal/api/handler.go
type Handler func(w http.ResponseWriter, r *http.Request) error

func (h Handler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    if err := h(w, r); err != nil {
        var apiErr *APIError
        if errors.As(err, &apiErr) {
            writeJSON(w, apiErr.Status, apiErr)
            return
        }
        writeJSON(w, 500, map[string]string{"error": "internal server error"})
        log.Printf("[handler] %v", err)
    }
}

// 使用
mux.Handle("/api/dashboard/overview", Handler(func(w http.ResponseWriter, r *http.Request) error {
    overview, err := svc.Overview(r.Context())
    if err != nil {
        return fmt.Errorf("overview: %w", err)
    }
    return writeJSON(w, 200, overview)
}))
```

### 4.3 `MonitorOverview` 返回 `map[string]interface{}`

```go
// store/sqlite.go:266
func (s *LogStore) MonitorOverview() (map[string]interface{}, error)
```

丢失类型安全，调用方不知道有哪些 key，IDE 无法补全。应返回具名结构体。

## 5. 错误处理与日志

### 5.1 错误包装不一致

代码中混用 `fmt.Errorf("xxx: %w", err)`（正确）和 `fmt.Errorf("xxx: %v", err)`（丢失 wrap）：

```go
// 正确（%w，支持 errors.Is/As）
return fmt.Errorf("build request: %w", err)              // handler.go:343

// 错误（%v，丢失 wrap 链）
log.Printf("[PANIC] %v\n%s", err, debug.Stack())         // main.go:823
```

建议统一用 `%w`，除非有特殊原因。

### 5.2 错误吞没

```go
// main.go:324
byModel, _ := logStore.ByModel()  // 错误被 _ 丢弃
```

```go
// store/intent_store.go:74
json.Unmarshal([]byte(r.Examples), &examples)  // 返回值被忽略
```

建议用 `_ =` 显式标注或处理错误。

### 5.3 缺少结构化日志

全部用 `log.Printf`，输出的是非结构化文本：

```go
log.Printf("[路由] → %s (model=%s)", provider.Name, modelName)
log.Printf("[启动] 数据库已就绪: %s", *dbPath)
```

**问题**：
- 无法按字段过滤（比如"查所有 model=deepseek 的路由日志"）
- 无法接入 ELK/Loki 等日志系统
- 没有日志级别（Debug/Info/Warn/Error）

**建议**：Go 1.21+ 内置 `log/slog`，无额外依赖：

```go
import "log/slog"

slog.Info("route selected",
    "provider", provider.Name,
    "model", modelName,
    "reason", "keyword match",
    "latency_ms", latency,
)
```

## 6. 可观测性缺口

### 6.1 缺少 Metrics

**当前**：`/api/resilience/stats` 和 `/api/routing/stats` 暴露了内部状态，但没有 Prometheus 指标。

**建议**：暴露 `/metrics` 端点，关键指标：

```
ragent_requests_total{provider, model, status}    counter
ragent_request_duration_seconds{provider}         histogram
ragent_tokens_total{provider, type(input|output)} counter
ragent_cost_usd_total{provider}                   counter
ragent_circuit_breaker_state{provider}            gauge(0=closed, 1=open, 2=half-open)
ragent_bulkhead_in_use                            gauge
ragent_rate_limit_remaining                       gauge
ragent_route_hits_total{strategy}                 counter
```

### 6.2 缺少 Tracing

当前 `requestIDMiddleware` 生成了 X-Request-ID，但没有接入 OpenTelemetry。在多级调用（客户端 → ragent → 上游供应商）中无法追踪全链路。

**建议**：集成 `go.opentelemetry.io/otel`，在 proxy handler 里创建 span：

```go
ctx, span := tracer.Start(r.Context(), "proxy.request", trace.WithAttributes(
    attribute.String("provider", provider.Name),
    attribute.String("model", modelName),
))
defer span.End()
```

### 6.3 健康检查可深化

当前 `/health` 只检查 DB ping 和供应商数量。建议分级：
- **Liveness**：进程存活（当前 `/healthz` 已实现）
- **Readiness**：DB + 至少一个供应商可用 + 路由引擎已初始化
- **Startup**：所有组件就绪（用于 Kubernetes startup probe）

## 8. 并发安全隐患（非致命但需关注）

### 8.1 `Proxy.debugProvider` 无锁读写

```go
// proxy/handler.go:228（读）
if p.debugProvider != "" { ... }

// proxy/handler.go:541（写）
p.debugProvider = name
```

`SetDebugProvider`（HTTP API 调用）和 `handleRequest`（代理请求）可能并发，存在数据竞争。`go build -race` 测试会报错。

**建议**：用 `atomic.Value` 或 `sync.RWMutex` 保护。

### 8.2 `Proxy.providers` / `Proxy.breakers` 运行时修改

```go
// proxy/handler.go:509
func (p *Proxy) AddProvider(cfg ProviderConfig) {
    p.providers[cfg.Name] = &cfg  // 无锁
    p.breakers[cfg.Name] = circuitbreaker.New(cbCfg)
}
```

注释说"仅用于初始化阶段"，但 `main.go` 没有运行时调用，目前安全。将来若要支持运行时增删供应商，必须加锁。

**建议**：用 `sync.RWMutex` 保护，或用 `atomic.Pointer` 替代 map（COW 模式）。

### 8.3 `handler.go:252-256` 运行时创建熔断器

```go
breaker := p.breakers[provider.Name]
if breaker == nil {
    breaker = circuitbreaker.New(cbCfg)
    p.breakers[provider.Name] = breaker  // 写 map 无锁
}
```

如果同一供应商的多个请求并发到达，会重复创建熔断器并覆盖。建议在 `NewProxy` 阶段为所有供应商预创建，或用 `sync.Map` + `LoadOrStore`。

## 9. 代码异味与小问题

### 9.1 重复代码

### 9.2 死代码 / 历史包袱

```go

type ProxyHealth struct {
    CCSwitchDBAvailable bool `json:"ccswitch_db_available"`  // 旧名字，已无意义
    ...

```

`CCSwitchDBAvailable` 字段从未被赋值（`main.go:444` 构造时没设置），属于历史遗留。应删除或重命名。

### 9.3 `main.go:388-389` 逻辑漏洞

```go
mux.HandleFunc("/api/proxy/current", func(w http.ResponseWriter, r *http.Request) {
    providers := p.ListProviders()
    if len(providers) == 0 { ... return }
    for _, prov := range providers {
        if prov.Enabled {
            writeJSON(w, http.StatusOK, ...)
            return
        }
    }
    // ⚠️ 如果有 providers 但都 disabled，这里不会写任何响应
    // 客户端会收到 200 + 空 body
})
```

建议在循环后补一个 `writeJSON(w, 200, map{... "is_valid": false})`。

### 9.4 CORS 仍然过于宽松

```go
// main.go:844-851
if allowedOrigins[origin] {
    w.Header().Set("Access-Control-Allow-Origin", origin)
} else {
    // 未知来源：仍允许但记录警告（开发阶段，生产应收紧）
    w.Header().Set("Access-Control-Allow-Origin", origin)
    log.Printf("[CORS] 未知来源: %s", origin)
}
```

`if/else` 两个分支做的事完全一样，等于无条件允许所有 Origin。注释说"开发阶段"，但生产部署时容易忘记收紧。建议用环境变量控制：

```go
if os.Getenv("CORS_ALLOW_ALL") == "true" {
    // 开发模式：允许所有
} else if allowedOrigins[origin] {
    // 生产模式：白名单
} else {
    // 拒绝
    w.WriteHeader(403)
    return
}
```

### 9.5 成本费率硬编码

```go
// proxy/handler.go:486-491
rates := map[string]rate{
    "deepseek": {0.27, 1.10},
    "claude":   {3.00, 15.00},
    ...
}
```

费率会变化（Claude 3.5 → 4 价格不同），且不同模型费率不同（claude-sonnet vs claude-opus）。应改为配置文件或数据库表。

### 9.6 韧性参数硬编码

```go
// main.go:154
GlobalRateLimit: 100, MaxConcurrentRequests: 50

// handler.go:153-154
cbCfg.FailureThreshold = 0.5
cbCfg.OpenTimeout = 30 * time.Second

// handler.go:347-348
retryCfg.MaxAttempts = 2

// handler.go:186
context.WithTimeout(r.Context(), 300*time.Second)
```

这些参数散落在 4 个文件里，无法整体调整。应集中到 `Config.Resilience` 结构体。

### 9.7 `AdapterFor` 和 `AdapterFactory` 双轨制

```go
// 包级函数
func AdapterFor(providerName string) ProviderAdapter { ... }

// 工厂类
type AdapterFactory struct { ... }
func (f *AdapterFactory) Get(providerName string) ProviderAdapter {
    if adapter, ok := f.adapters[providerName]; ok { return adapter }
    return AdapterFor(providerName)  // 回退到包级函数
}
```

两套机制并存，容易混淆。建议统一：要么只有工厂，要么只有包级函数 + 注册表。

### 9.8 `routing.Intents.go` 的 `DefaultIntents` 与 `store/intent_store.go` 的 `SeedDefaults` 重复

两个地方各自硬编码了一份默认意图，内容几乎相同。如果改了一处忘了改另一处，会导致 DB 数据和 fallback 数据不一致。应让 `DefaultIntents` 调用 `SeedDefaults` 的数据源，或反之。

## 11. 值得保留的设计亮点

批评之外，这些设计做得很好，重构时**不要破坏**：

1. **韧性五件套**（限流/舱壁/熔断/重试/超时）的分层设计和执行顺序，符合业界最佳实践
2. **`ProviderAdapter` 接口**抽象了协议差异，新增协议只需实现接口
3. **`RouteMatcher` 接口**让路由引擎可插拔
4. **`HybridRouter` 的渐进式降级**策略（关键词 → Embedding → LLM → 兜底）
5. **`EmbeddingCache` 的跨 chunk SSE 解析**（`TokenTracker` 的 `bufio.Scanner` + `bytes.Buffer` 组合）
6. **注释质量**：算法复杂度、ASCII 流程图、设计权衡说明，在 Go 项目中罕见地详尽
7. **优雅关闭** + **健康检查分级**（liveness/readiness）
