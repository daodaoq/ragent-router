# RAgent Router 产品演进方向建议

> 当前状态：基础设施和代码质量已夯实（韧性引擎、路由引擎、Token 计量、Dashboard、供应商管理），致命 Bug 和架构问题均已修复。
> 下一步：从「好用的个人工具」向「不可替代的 AI 网关平台」演进。

---

## Tier 1 — 最高 ROI（1-2 周可交付）

这三个功能解决用户**最痛的痛点**，投入产出比最大，且架构上已有铺垫。

### 1.1 语义缓存（Semantic Cache）

**为什么最优先：** 这是唯一一个**不调用 API 就能完成请求**的功能，直接省钱 30-50%。你已经有 embedding 向量化能力，只需加一层缓存判断。

**用户痛点：**
- 日常开发中 30-40% 的请求是重复或高度相似的（"这个函数什么意思"、"帮我格式化这段代码"）
- 每次都花 Token 调用 API，浪费成本
- 简单问题 2-3 秒响应，有了缓存可以 <100ms

**实现思路：**
```
请求 → Embedding 向量化 → 与缓存向量库比对
  ├─ 余弦相似度 > 阈值(0.92) → 直接返回缓存结果（不走 API）
  └─ 相似度 < 阈值 → 正常路由转发 → 结果写入缓存
```

**技术要点：**
- 利用已有的 `embedding.go` 的 EmbeddingCache 基础设施
- 缓存存储：SQLite 新表 `semantic_cache(id, prompt_embedding BLOB, response TEXT, provider, model, tokens, created_at, ttl)`
- 相似度阈值可配置（默认 0.92），过低会返回不相关答案，过高命中率低
- TTL 过期策略：技术类答案 24h，创意类答案不缓存
- Cache 命中时在 SSE 流里加一个 `event: cache_hit` 事件，让前端知道是缓存结果
- Dashboard 新增"缓存命中率"指标卡片

**预估收益：** 30-50% 的请求不走 API，月度费用直降。

---

### 1.2 预算管控与自动降级（Budget Guard）

**为什么优先：** 让用户敢放心用。没有预算控制，用户永远在焦虑"这个月会不会超"，宁可少用也不敢放开用。有了预算控制 + 自动降级，反而用得更多。

**用户痛点：**
- "不知道这个月花了多少，怕超预算"
- "月初猛用，月末心疼"
- "团队里有人滥用昂贵模型"

**实现思路：**
```
1. 预算设置 API：
   POST /api/budget/set
   { "monthly_limit": 50, "warning_threshold": 0.7, "fallback_provider": "DeepSeek" }

2. 每次请求前检查：
   当前月度累计费用 / 月度预算 = 使用率
   ├─ < 70% → 正常路由（可以用昂贵模型）
   ├─ 70%-90% → 只路由到廉价模型（自动降级）
   ├─ > 90% → 只允许缓存命中或最廉价模型
   └─ > 100% → 返回 429 Too Many Requests + 降级提示

3. Dashboard 新增：
   - 预算进度条（本月已用 / 预算上限）
   - 预算状态标签（正常/警告/降级中/已耗尽）
   - 降级触发历史
```

**技术要点：**
- 预算数据存 SQLite：`budget_settings(id, monthly_limit, warning_pct, hard_pct, fallback_provider, updated_at)`
- 月度累计从已有的 `request_logs` 表聚合
- 降级逻辑插入路由引擎：在 `HybridRouter.Match` 之前检查预算状态
- 降级不是断服务，是自动切廉价模型——用户体感是"响应质量降了但还能用"
- 预算重置按自然月，cron 在每月 1 号自动清零累计

**关键设计决策：** 降级策略应该是**渐进式**而非一刀切：
- 阶段 1（70%）：复杂任务仍可走 Claude，简单任务走 DeepSeek
- 阶段 2（90%）：所有请求走 DeepSeek
- 阶段 3（100%）：只允许缓存命中 + DeepSeek-mini

---

### 1.3 API Key 池管理（Key Pool）

**为什么优先：** 实际使用中的刚需。单个 API Key 有速率限制（DeepSeek 30 RPM），多人共用或高频调用时经常被限流。

**用户痛点：**
- 单 Key 被限流后请求排队等待
- 不同供应商的 Key 生命周期不同，有的过期了才发现
- 多个 Key 手动管理混乱

**实现思路：**
```
1. Key Pool 数据模型：
   provider_keys(id, provider_name, api_key, weight, status, 
                 rate_limit_rpm, last_used, last_error, created_at)

2. Key 选择策略：
   ├─ Round-Robin：均匀轮转（默认）
   ├─ Weighted：按权重分配（付费 Key 权重高）
   └─ Least-Used：最少使用优先（延长冷却期）

3. Key 健康监控：
   - 连续 3 次 401 → 标记 invalid，自动移出池
   - 连续 5 次 429 → 标记 rate_limited，冷却 60s 后重新尝试
   - Key 过期前 7 天 → Dashboard 告警

4. API：
   POST   /api/keys/add        添加 Key
   DELETE /api/keys/{id}       移除 Key
   GET    /api/keys/status     各 Key 健康状态
```

**技术要点：**
- Key 选择在 `handler.go` 的 `doUpstreamRequest` 之前执行
- 当前 `ProviderConfig` 只支持单个 `APIKey`，需要改为 `[]KeyEntry`
- Key 加密存储（SQLite 里存 AES-256 加密后的值，运行时解密）
- 与已有的限流器联动：令牌桶的 rate 按 Key 数量 × 单 Key RPM 计算

## Tier 2 — 高差异化（2-4 周交付）

这三个功能让产品从「好用」变成「不可替代」。

### 2.1 请求中间件管线（Middleware Pipeline）

**为什么做：** 这是让 RAgent Router 从"代理"变成"平台"的关键。中间件管线让第三方（或用户自己）能在请求流转过程中插入自定义逻辑。

**用例场景：**
- **PII 脱敏**：自动检测并替换手机号、身份证号、邮箱等敏感信息
- **Prompt 增强**：自动附加 system prompt（"请用中文回答"、"请给出代码示例"）
- **内容安全**：检测有害内容，拦截违规请求
- **审计日志**：记录完整的请求/响应内容用于合规
- **自定义路由**：根据业务规则决定走哪个模型

**实现思路：**
```go
// 中间件接口
type RequestMiddleware interface {
    Name() string
    // 请求阶段：在转发到上游之前执行
    ProcessRequest(ctx context.Context, req *ProxyRequest) (*ProxyRequest, error)
    // 响应阶段：在返回给客户端之前执行（可选）
    ProcessResponse(ctx context.Context, resp *ProxyResponse) (*ProxyResponse, error)
}

// 管线执行顺序
func (p *Pipeline) Execute(ctx, req) {
    for _, m := range p.middlewares {
        req, err = m.ProcessRequest(ctx, req)
        if err != nil { break }  // 某个中间件可以拦截请求
    }
    resp := doUpstreamRequest(ctx, req)
    for _, m := range p.middlewares {
        resp, _ = m.ProcessResponse(ctx, resp)  // 响应阶段不拦截
    }
    return resp
}
```

**技术要点：**
- 中间件注册表：`map[string]RequestMiddleware`，通过 YAML 配置启用/禁用
- 执行顺序按配置文件的排列顺序
- 每个中间件可以选择性实现 `ProcessResponse`（大部分只处理请求）
- SSE 流式响应的中间件处理较复杂，建议只在 `ProcessRequest` 阶段拦截
- 内置 3 个默认中间件：PII Detection、Prompt Enrichment、Content Safety

### 2.2 自适应路由（Adaptive Routing）

**为什么做：** 当前路由规则是静态的——人工定义"architecture 相关走 Claude"。但实际使用中，哪个模型擅长什么应该是**数据驱动**的，而不是靠直觉。

**核心思路：** 从历史请求中学习，自动调优路由策略。

```
当前：规则匹配 → 固定供应商
未来：规则匹配 → 候选供应商列表 → 按历史成功率排序 → 选最优
```

**实现思路：**
```
1. 效果反馈收集：
   每次请求完成后，用户可以标记"满意/不满意"（前端加个拇指按钮）
   或者自动指标：响应长度、是否包含代码、是否被中断

2. 统计聚合（定时任务，每小时）：
   provider_performance(provider, intent_category, 
     avg_latency, avg_tokens, avg_cost, satisfaction_rate, sample_count)

3. 路由决策调整：
   同一意图类别下，如果 DeepSeek 的满意度 > Claude，
   自动把该类别的首选调整为 DeepSeek
   调整幅度有上限（每周最多调整 20%），防止震荡

4. Dashboard 新增：
   - "模型效果对比"页面：按意图类别展示各模型的表现
   - "路由策略演变"时间线：展示自适应调整的历史记录
```

**技术要点：**
- 新 SQLite 表：`response_feedback(id, request_log_id, rating, created_at)`
- 统计聚合用 SQL window function，每小时一次
- 调整策略写回 `routing_rules` 表，但不覆盖用户手动定义的规则
- 需要一个"学习开关"：用户可以选择"仅用静态规则"或"允许自适应"

---

### 2.3 模型效果分析（Model Performance Analytics）

**为什么做：** 让用户做出明智的模型选择。当前 Dashboard 只有"谁被调用了多少次"，没有"谁表现更好"。有了效果分析，用户能知道"GPT-4 虽然贵但代码生成质量远高于 DeepSeek"或"DeepSeek 回答简单问题性价比极高"。

**实现思路：**
```
Dashboard 新页面 "模型效果分析"：

┌─────────────────────────────────────┐
│  模型效果三角：质量 × 速度 × 成本    │
│                                     │
│  Claude ●                           │  质量高、成本高
│          ● DeepSeek                  │  质量中、成本低
│       ● GPT-4                       │  质量高、成本极高
│                                     │
│  按意图类别切换：                     │
│  [代码生成] [架构设计] [简单问答]      │
│                                     │
│  每个类别下：                         │
│  - 平均响应延迟                       │
│  - 平均 Token 消耗                    │
│  - 平均单次成本                       │
│  - 用户满意度（如有反馈）              │
│  - 样本量                            │
└─────────────────────────────────────┘
```

**技术要点：**
- 数据来源就是已有的 `request_logs` 表，按 `provider + model` 聚合
- 新增 API：`GET /api/analytics/model-performance?category=code_generation`
- 前端用 Recharts 画散点图（x=成本, y=满意度, 点大小=样本量）
- 与 2.2 自适应路由共享同一套 `provider_performance` 统计表

## Tier 3 — 战略布局（4-8 周交付）

这三个功能解锁**商业模式**和**平台生态**。

### 3.1 多模型编排（Multi-Model Orchestration）

**为什么做战略级：** 这是市场上几乎没有产品做到的功能。不是"选一个模型"，而是"同时调多个模型，择优或融合返回"。这是一个真正不可替代的差异化能力。

**用例场景：**
- **择优模式**：同一问题同时发给 Claude + DeepSeek，对比后返回更好的答案
- **融合模式**：Claude 写架构，DeepSeek 补细节，融合成一个完整回答
- **验证模式**：DeepSeek 生成代码，Claude 审查代码质量，返回审查结果
- **冗余模式**：Claude 熔断时自动用 GPT-4 的结果，保证不空返

**实现思路：**
```
1. 编排策略定义：
   orchestration_policies(id, name, type, providers, 
     prompt_template, selection_criteria)

   type 枹举: "best_of_n" | "merge" | "review" | "fallback_chain"

2. 执行引擎：
   func ExecuteOrchestration(ctx, req, policy) {
       switch policy.Type {
       case "best_of_n":
           // 并行调 N 个模型，用 LLM 评判选最优
           responses := parallelCall(req, policy.Providers)
           return judgeBest(responses)
       case "merge":
           // 串行调，每次带上前一个的回答作为上下文
           return sequentialMerge(req, policy.Providers)
       case "review":
           // 先生成，再审查，返回审查后版本
           draft := call(req, policy.Providers[0])
           return call(reviewPrompt(draft), policy.Providers[1])
       }
   }

3. SSE 流式处理：
   - best_of_n：先并行调，收集完后一次性返回最优结果
   - merge/review：串行流式，每个阶段用 `event: phase_change` 标记切换
```

**技术要点：**
- 并行调用用 goroutine + `errgroup`，已有 bulkhead 可控制并发数
- 评判者可以是用户指定的 LLM（"让 Claude 来评判 Claude vs DeepSeek 的回答"）
- 成本计算：best_of_n 模式费用 = N × 单次费用，需要在 UI 里明确提示
- 新增 SSE event 类型：`phase_change`、`model_switch`、`judge_result`

---

### 3.2 Prompt 模板库（Prompt Template Library）

**为什么做：** 降低使用门槛。不是每个人都会写好的 Prompt，模板库让新手也能获得高质量回答。同时也是社区/生态的起点——用户可以分享模板。

**实现思路：**
```
1. 模板数据模型：
   prompt_templates(id, name, category, template_text, 
     default_provider, default_model, variables, author, 
     version, created_at, usage_count, rating)

   variables 示例: ["language", "framework", "task_type"]
   template_text 示例: "请用 {{language}} 为 {{framework}} 实现 {{task_type}}"

2. 模板应用流程：
   用户选择模板 → 填写变量 → 自动组装 Prompt → 发送到路由引擎
   
3. 模板市场：
   - 内置 10 个精选模板（代码生成、架构设计、Bug 分析、文档编写等）
   - 用户可创建/保存自己的模板
   - 模板评分和使用量排行

4. A/B 测试：
   同一个任务，用不同模板发给同一模型，对比效果
   记录哪个模板的满意度更高
```

**技术要点：**
- 变量替换用 Go `text/template`
- 模板存储：SQLite `prompt_templates` 表
- 前端新增"模板浏览器"页面
- 模板可以指定推荐供应商（"这个模板最适合 Claude"）

---

### 3.3 多租户与团队管理（Multi-Tenant）

**为什么做战略级：** 从个人工具变成团队基础设施。一旦有了团队管理，这个产品就不再是一个人用的小工具，而是公司内部的平台。

**实现思路：**
```
1. 数据模型：
   teams(id, name, monthly_budget, created_at)
   team_members(team_id, user_id, role)  -- role: admin / member
   team_api_keys(team_id, provider, api_key, key_pool_id)
   team_quota(team_id, monthly_limit, current_usage)

2. 团队隔离：
   - 每个团队有独立的 API Key 池
   - 每个团队有独立的预算和配额
   - 请求日志按团队隔离
   - 路由规则可以团队级别自定义

3. 管理功能：
   - 团队管理员可以查看成员用量
   - 可以设置成员级别的配额上限
   - 可以审批昂贵模型的调用（"超过 $0.5/次的请求需要审批"）

4. API：
   POST   /api/teams/create
   POST   /api/teams/{id}/members/add
   GET    /api/teams/{id}/usage
   GET    /api/teams/{id}/members
```

**技术要点：**
- 认证层：从当前的静态 API Key 认证升级为 JWT + team_id
- 所有现有表加 `team_id` 字段做隔离
- 前端新增"团队管理"页面
- 审批流可以用简单的异步队列（SQLite 足够）

## 功能之间的依赖关系

```
语义缓存 ─────→ 自适应路由（缓存命中率是路由决策的输入之一）
预算管控 ─────→ API Key 池（预算降级时需要知道 Key 池的剩余容量）
中间件管线 ──→ 多模型编排（编排的择优判断可以是一个中间件）
模型效果分析 ← 自适应路由（共享 provider_performance 统计表）
Prompt 模板 ──→ 中间件管线（模板变量替换是请求预处理的一部分）
多租户 ───────→ 预算管控（团队级别的预算需要多租户隔离）
```

**建议执行顺序：**
1. 语义缓存 → 预算管控 → API Key 池（Tier 1 串行，互相不依赖）
2. 中间件管线 → 自适应路由 + 模型效果分析（Tier 2，管线先做，路由和分析可并行）
3. 多模型编排 → Prompt 模板 → 多租户（Tier 3 串行，编排先做，模板依赖编排，多租户依赖所有前置）

## 我的推荐：从哪里开始？

**如果只选一个：语义缓存。** 

理由：
- ROI 最高——直接省钱，用户体感最明显
- 架构铺垫最好——你已经有 embedding 能力，只需加缓存层
- 1 周可交付
- 为后续自适应路由提供数据基础

**如果选两个：语义缓存 + 预算管控。**

理由：
- 缓存省钱，预算控费——双管齐下解决成本焦虑
- 预算管控让用户敢放开用，反而提升产品粘性
- 两个功能互不依赖，可以并行开发

**如果选三个：语义缓存 + 预算管控 + API Key 池。**

理由：
- Tier 1 全做完后，产品在「成本控制」这个维度上就完整了
- 之后进入 Tier 2 时心态从容——核心痛点已解决，可以做差异化

---

## 不建议现在做的（容易踩坑）

1. **开源社区/贡献者体系** — 产品还没到让人主动贡献的阶段，先让核心功能稳定
2. **移动端 App** — AI Gateway 的核心场景是桌面开发，移动端需求弱
3. **多语言 SDK** — 当前只有 Go 后端 + Electron 前端，SDK 需要用户基数支撑
4. **付费 SaaS 版** — 先做出差异化功能（Tier 2），再考虑商业模式
5. **实时协作/多人聊天** — 这不是 Gateway 的职责，偏离核心定位
