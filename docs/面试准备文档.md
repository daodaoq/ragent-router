## 三、模块逐一讲解（面试话术）

### 模块 1：令牌桶限流器（ratelimit/tokenbucket.go）

**面试开场**：
> "令牌桶算法本身不复杂，但我在热路径上做了一个优化叫 Lazy Refill。"

**核心概念**：
- 常规实现每次 Allow() 都调 time.Now() 计算时间差 → 补充 Token
- 我的实现：Token 够用时只做 Mutex + 整数递减，不碰系统调用
- 只有桶空了才走冷路径：time.Since + 除法算补充量
- Benchmark 结果：28ns/op，零内存分配

**常见追问**：

> Q: "为什么不直接用 golang.org/x/time/rate？"

"标准库的 rate.Limiter 每次 Allow() 都调 time.Now()，高 QPS 下有 20-30% 的性能差异。我测过——标准库的 Allow 大约 35-40ns，我的 Lazy Refill 版本是 28ns。差距来源于省掉了一次系统调用。代价是我的实现不支持动态调整 rate（标准库支持），但 API 网关场景下 rate 配置后不变，所以这不是问题。"

> Q: "28ns 怎么测出来的？"

"Go benchmark：BenchmarkTokenBucket_Allow-32 跑在 i9-13900HX 上，GOMAXPROCS=32。关键指标是 0 B/op 0 allocs/op——热路径完全没有内存分配，也就没有 GC 压力。"

> Q: "lastRefill 为什么用 Add 而不是直接设 time.Now()？"

"如果直接设 time.Now()，每次桶空后都会把 lastRefill 推进到当前时间，意味着「这段时间没请求」被当成了「这段时间在匀速消耗 Token」。比如 rate=100/s，空闲 5 秒后再来请求——按 time.Now() 会一次性补满 100 个 Token。而按 Add 方式，5 秒只能补 500 个——精确按 rate 来。前者会导致长时间空闲后出现不符合 rate 的突发流量。"

### 模块 2：分片锁存储（ratelimit/sharded_store.go）

**面试开场**：
> "限流器的状态需要按 key（比如用户 ID 或 API key）存储，避免不同用户的请求互相影响。我实现了一个基于分片锁的并发安全 KV 存储。"

**核心概念**：
- 2048 个分片，FNV-64a 哈希路由
- 每个分片独立的 sync.RWMutex——不同分片间的操作完全并行
- Load 路径使用 Double-Checked Locking：先读锁查 → 没找到再写锁 + 二次确认
- 100 个 goroutine 并发 Load 同一 key，builder 函数仅调用 1 次（验证通过）

**常见追问**：

> Q: "为什么是 2048 个分片？"

"经验值。分片数至少要是预期并发 goroutine 的量级——2048 在 1000 并发下，每个分片平均 0.5 个 goroutine 在竞争。太小了锁冲突，太大了浪费内存（每个分片 ~100 字节，2048 个也只有 200KB）。用 Benchmark 验证过，1024 和 2048 性能差异不大，但 512 在高并发下已经有可观察到的锁竞争。"

> Q: "Double-Checked Locking 在 Go 里真的需要吗？不需要。sync.Map 不是更好？"

"Double-Checked Locking 解决的是 TOCTOU——两次检查之间 key 可能被创建。不写第二次检查会导致 key 被创建两次，先创建的值泄漏。sync.Map 适合写一次读多次的场景（如缓存配置），但它的 LoadOrStore 语义要求 builder 在锁外调用——如果 builder 很重（如创建令牌桶），多个 goroutine 会同时执行 builder，只有一个的结果被保留。我的实现中 builder 只在写锁内执行且二次确认后才存储，保证只会执行一次。"

> Q: "为什么不直接用 sync.Map？"

"sync.Map 适合 key 集合稳定、写一次读多次的场景。限流器存储的特点是 key 频繁增删（TTL 驱逐），且 Load 结果的 value 需要强一致性。sync.Map 在这种场景下 API 不够直观，且无法嵌入自定义的 lastAccess 追踪。分片 RWMutex + 普通 map 更可控。"

### 模块 3：熔断器（circuitbreaker/breaker.go）

**面试开场**：
> "这是项目里最有技术深度的组件。它实现了一个三态状态机，中间有半开状态作为恢复的过渡。"

**核心概念**：
- 三个状态：Closed（正常）→ Open（熔断，快速失败）→ HalfOpen（探测恢复）
- 失败率通过滑动窗口（N 个时间桶的环形缓冲区）统计
- 半开状态只允许有限探测请求——成功则闭合，失败则重新熔断
- 每个上游供应商独立一个熔断器实例——一个挂了不影响其他

**常见追问**：

> Q: "滑动窗口和固定窗口有什么区别？"

"固定窗口（如'过去 10 秒'）有边界效应——窗口切换瞬间，错误率会从 0 跳到 100% 或反过来。比如第 9.9 秒大量失败，第 10.1 秒窗口切换后错误率骤降，熔断器可能过早恢复。滑动窗口把时间分成 N 个桶，最旧的桶逐步移出——错误率变化平滑。代价是多一点内存（我的实现中 10 个桶，可以忽略不计）。"

> Q: "半开状态为什么只放一个请求？"

"为了防止流量尖峰——如果服务刚恢复还不稳定，放 100 个请求进去可能又把它打崩了。1 个探测请求的代价是「慢了一点点」（需要等这个请求完成），但好处是「如果它失败了，只打崩了 1 个请求」。这是保守策略，对 API 网关非常合适。"

> Q: "状态切换需要加锁吗？为什么不用 atomic？"

"状态切换涉及清理半开计数器、重置失败时间等多个字段，需要原子性。Mutex 是最清晰的保证方式。读多写少（99% 的时间在 Closed 状态直接返回），Go 的 Mutex 在这种非竞争场景下很快（~25ns），不需要 atomic 的复杂性。"

> Q: "跟 Hystrix（Java）/Sentinel（Java）的区别？"

"设计理念一致：三态状态机 + 滑动窗口统计。区别在于语言和复杂度——Hystrix 有线程池隔离、信号量隔离、请求缓存等全套功能，但非常重，启动一个 HystrixCommand 开销不小。我的实现是轻量级 Go 版本——只用 Mutex + 环形数组，几百行代码，零外部依赖。适合嵌入到任何 Go 项目中。"

### 模块 4：Jitter 重试（retry/backoff.go）

**面试开场**：
> "重试本身不复杂。有价值的是 Jitter——为什么重试需要加随机数。"

**核心概念**：
- 指数退避：cap = min(maxDelay, baseDelay × 2^attempt)
- 三种 Jitter 策略，对应不同场景
- 没有 Jitter 时的惊群效应：N 个客户端同时被限流 → 同时等 1s → 同时重试 → 同时被限流

**常见追问**：

> Q: "为什么选择 Full Jitter 作为默认值？"

"AWS Architecture Blog 有一篇经典文章分析了各种 Jitter 策略。Full Jitter 在客户端数量多时分散效果最好——因为随机范围最大，不同客户端的重试时机几乎不重叠。它的代价是平均等待时间比 Equal Jitter 短（只有 cap/2），在某些需要更激进的场景可能不合适。Equal Jitter 保留了更多的确定性时序，适合定时精度要求高的场景。"

> Q: "Decorrelated Jitter 的应用场景？"

"多节点部署时。如果多个节点同时被限流，Full Jitter 仍然会有少量重叠（每个节点的随机数可能碰巧接近）。Decorrelated Jitter 的每次延迟跟前一次无关，跨节点重试时序趋近于独立的随机过程——这在概率上更优。不过实现复杂一点，所以我提供了三种可选。"

### 模块 5：舱壁隔离（bulkhead/bulkhead.go）

**面试开场**：
> "舱壁的核心思想是：一个慢依赖不能耗尽所有资源。我的实现非常简单——一个带缓冲的 channel 作为计数信号量。"

**核心概念**：
- channel 缓冲大小 = 最大并发数
- 非阻塞发送 → 快速失败（不排队）
- 请求完成后从 channel 取出（释放槽位）

**常见追问**：

> Q: "为什么不排队而是直接拒绝？"

"排队的问题是请求堆积——如果上游慢了，100 个请求排队等着，每个在等上一个完成 → 内存占用飙升 → OOM。快速失败让调用方立刻知道无法处理，可以做降级或换一个 endpoint。这对 API 网关来说比排队安全。"

> Q: "跟令牌桶的限流有什么区别？"

"令牌桶控制的是'速率'（每秒多少请求），舱壁控制的是'并发度'（同一时刻多少请求在进行中）。两者互补：限流器防止调用太快（频率），舱壁防止同时太多（并发）。举个例子：限流器设 QPS=100，但每个请求耗时 10s→同时在途的可能有 1000 个请求，限流器管不了这个。舱壁设并发=10→同时在途的永远不超过 10 个。"

### 模块 6：级联超时（timeout/cascade.go）

**面试开场**：
> "级联超时解决一个常见 bug：每个子操作设了超时，但因为父 context 没有 deadline，实际执行时间远超预期。"

**核心概念**：
- Cascading 创建的子 deadline = min(父 deadline, 自己的 timeout)
- 这就是 Go context 语义的自然扩展
- 防止"超时逃逸"：重试循环 3 次 × 10s = 实际上跑了 30s

**常见追问**：

> Q: "Go 的 context.WithTimeout 本身不就有级联效果吗？"

"对，context.WithTimeout 自己就会取 min。我这层包装的价值在于语义清晰——Cascading 这个名字明确表达了'我服从父 deadline'的意图。另外加了 Remaining 辅助函数来在调用链中间检查还剩多少时间。不是什么重活，但在调用链很深时让意图更明确。"

## 四、常见面试追问汇总

### 项目整体相关

> Q: "为什么用 Go 写这个项目？"

"两个原因。第一，Go 的并发模型（goroutine + channel）天然适合 API 网关这种 IO 密集型场景——大量网络请求、连接复用、流式传输。Python 的 asyncio 也能做，但 Go 的 goroutine 调度比 asyncio 的事件循环更适合 CPU 密集和 IO 密集混合的场景。第二，我秋招目标就是 Go 后端岗，这个项目是我对 Go 并发编程的系统性实践。"

> Q: "这个项目有什么实际应用？"

"我就是用它来代理我日常 Claude Code 的 API 调用的。虽然现在 Go 版本刚写完还没有跑满生产量，但 Python 原型版本已经在用——日均处理 1000+ 请求。Go 重写的目标是把 Python 版本的容错短板补上，同时作为秋招项目展示我的 Go 工程能力。"

> Q: "如果让你给项目打分，你觉得哪些地方可以改进？"

"三点。一是熔断器的滑动窗口目前是固定桶数，理想情况下应该自适应——流量低时少开桶节约内存，流量高时多开桶保证精度。二是路由引擎目前是关键词匹配，下一步想用 embedding 相似度来提升匹配准确率。三是缺少集成测试——目前只有单元测试和 demo 场景，没有模拟真实上游故障的端到端测试。"

### 并发相关

> Q: "Go 的 map 不是并发安全的，你怎么处理的？"

"三个层面：(1) 分片存储每个分片有独立的 sync.RWMutex；(2) Load 路径的读路径用 RLock，对大部分请求零竞争；(3) sync.Mutex 的 Lock/Unlock 提供 happens-before 语义，保证写入对其他 goroutine 可见。不需要 atomic 因为 Mutex 已经提供了足够的内存屏障。"

> Q: "sync.Mutex 和 sync.RWMutex 的区别？什么时候用哪个？"

"RWMutex 允许并发读——多个 goroutine 同时持读锁，不互相阻塞。写锁互斥，且等待所有读锁释放。适合读多写少的场景。普通 Mutex 读写都互斥——无论读写都只能一个人进。RWMutex 比 Mutex 稍微重一点（多维护一个 readerCount），所以在'读写比例接近 1:1'时直接用 Mutex 反而更快。我的场景里 Load 远多于 Store（>100:1），RWMutex 是正确选择。"

> Q: "goroutine 泄漏怎么排查？"

"用 pprof 的 goroutine profile——看哪些 goroutine 在阻塞，阻塞在哪个函数，是不是有 channel 操作在等待永远不来的信号。这个项目的舱壁用的是无缓冲 signal channel——如果忘记 defer 释放槽位，goroutine 会永久泄漏。所以我在 Execute 里用了 defer func() { <-b.sem }()，确保无论 fn 成功还是 panic 都会释放。"
