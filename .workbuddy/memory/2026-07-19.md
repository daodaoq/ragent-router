# 2026-07-19 工作日志

## 项目分析：RAgent Router 改进建议

对 ragent-router 项目（Go 后端 + Electron/React 前端的 AI API 智能网关）做了全量代码审查，生成改进建议报告 `docs/项目改进建议报告.md`。

### 关键发现
- **P0 紧急**：后端无认证、CORS 全开；请求日志 Prompt 永远为空（`main.go:160` 传入空字符串）；`proxyErr` 变量声明未赋值；`/api/proxy/activate` 是空操作；`/api/monitor/overview` 返回硬编码 0；前端 16 处硬编码 URL、无 ErrorBoundary、Dashboard 死代码链。
- **P1 高优**：无结构化日志、无配置文件（韧性参数/成本费率全硬编码）；`EmbeddingCache.evictLoop` goroutine 泄漏；proxy/store/handler 零测试；前端 111 处 `lang==="zh"?` 绕开 i18next；无 ESLint/Prettier/CI。
- **P2 中优**：无 metrics/tracing；健康检查只返回静态 ok；SSE 跨 chunk 解析缺陷；前端巨型组件（IntentPanel 599行/Settings 482行）；无 react-query；9 处 `.catch(()=>{})` 静默吞错。
- **P3 细节**：重复造轮子（自实现 strings.Contains/Join）、server.exe 误提交、a11y 缺失、electron/main.js 是 .js 非 .ts、README 与代码不一致（声称 Go 实际 spawn Python uvicorn）。

### 亮点
韧性引擎设计扎实（令牌桶 28ns/op、三态熔断、级联超时）；后端注释充分含算法复杂度分析；前端 TS strict + Zustand + CSS 变量主题方案方向正确。

## 后端深度审查（逐行）

对 backend/ 全部 20 个 Go 文件逐行审查，生成 `docs/后端深度审查报告.md`。相比首次审查，新发现多个严重 Bug：

### 新发现的严重 Bug（P0）
1. **关键词路由 Provider 大小写不匹配**（`matcher.go:211` Provider:"claude" 小写 vs `main.go:64` providerMap key="Claude" 大写）→ 关键词路由层完全失效，所有命中关键词的请求穿透到默认回退 DeepSeek。"架构设计该用 Claude"这个核心卖点失效。
2. **OpenAI/DeepSeek 的 SSE Token 统计失败**（`token_tracker.go:197-208` 只读 Anthropic 的 input_tokens/output_tokens 字段名，不兼容 OpenAI 的 prompt_tokens/completion_tokens）→ DeepSeek/OpenAI 供应商的成本统计全为 0。
3. **SSE 流式传输中重试导致响应混乱**（`handler.go:249-253` retry.Do 包裹 doUpstreamRequest，但 WriteHeader(200) 后 io.Copy 失败会触发重试，导致二次 WriteHeader panic）。
4. **Match 读 r.intents 无锁**（`hybrid_engine.go:276`）与 ReloadIntents 写 r.intents 存在数据竞争。

### 新发现的设计缺陷
- MiniMax 适配器选择错误（`adapter.go:178` 只识别 openai/deepseek，MiniMax 被当 Anthropic 原生格式）
- OpenAI 适配器丢失 system prompt（`adapter.go:96` Anthropic top-level system 未翻译）
- 模型匹配逻辑反向（`matcher.go:133` strings.Contains(prov.Name, model) 应为 Contains(model, prov.Name)）
- SQLite 单连接导致读写互相阻塞（`sqlite.go:43` SetMaxOpenConns(1)）
- Delete 意图无事务、CompactPrompt 按字节截断中文乱码
- 熔断器测试用 t.Logf 而非 t.Fatalf（`demo_test.go:92` 测试无效）

## 架构与设计模式审查（第三轮）

用户反馈已修复前两轮 P0 问题后，重新通读全部 20 个 Go 文件，确认修复情况良好（新增 bulkhead 舱壁、LLMIntentClassifier、intents.go、hybrid_engine_test.go；SSE 重试用 ErrStreamStarted 标识；关键词大小写已统一；Token 兼容 OpenAI 字段；Delete 用事务；CompactPrompt 按 rune；中间件补齐 recovery/requestID/auth；monitor 真查询 DB；SQLite 开 4 连接）。

从架构演进视角生成 `docs/架构与设计模式改进报告.md`，核心发现：

### 架构层面
- `main.go` 944 行承担 5+ 种职责（HTTP 路由/中间件/配置加载/DTO 转换/工具函数），违反 SRP，建议拆分到 `internal/api/`
- `routing` 包 import `proxy.ProviderConfig` 形成双向耦合（`proxy` 通过 RouteMatcher 接口调 routing，routing 又依赖 proxy 具体类型），建议提取 `internal/provider` 包

### 设计模式机会
- **策略模式深化**：HybridRouter 内 3 个策略是硬编码 if 链，新增策略要改 Match 方法体。建议抽象 `RouteStrategy` 接口 + 策略链
- **责任链模式**：韧性层（限流/舱壁/熔断/重试/超时）是嵌套调用，建议抽象成 Middleware 链可组合可配置
- **观察者模式**：OnRequestLog 是单回调，建议改为 `[]RequestObserver` 支持多观察者（日志/指标/告警）
- **仓储模式**：LogStore 混合 CRUD + 业务查询 + 返回 map[string]interface{}，建议分离 Repository 接口 + DashboardService
- **选项模式**：NewProxy(cfg) 配置结构体会膨胀，建议 Functional Options

### 可扩展性
- 新增供应商需改 4 处分散代码（loadProviders/AdapterFor/estimateCost/DefaultRules），无一致性保证，建议聚合为 Provider 抽象 + Registry
- 切换存储后端需重写整个 LogStore，未定义接口
- 只有意图支持热重载，供应商/规则/韧性参数修改需重启

### 其他
- debugProvider 字段无锁读写（handler.go:228 读 vs 541 写），数据竞争
- CompactPrompt 在 sqlite.go 和 intent_store.go 重复定义
- CORS if/else 两个分支做的事完全一样（main.go:844-851），等于无条件允许
- 全部用 log.Printf，建议升级到 log/slog（go.mod 已是 1.22）
- 无 metrics/tracing，proxy/store/handler 零测试

## 产品演进方向建议（第四轮）

用户确认前三轮所有问题已修复，询问下一步方向。从产品竞争力角度生成 `docs/产品演进方向建议.md`，按 Tier 分 3 层：

- **Tier 1（1-2 周）**：语义缓存（省 30-50% API 费）、预算管控与自动降级（月度上限+渐进降级）、API Key 池管理（多 Key 轮转突破速率限制）
- **Tier 2（2-4 周）**：请求中间件管线（PII脱敏/内容安全/可插拔预处理链）、自适应路由（历史数据驱动路由策略调优）、模型效果分析（延迟/质量/成本三角对比）
- **Tier 3（4-8 周）**：多模型编排（并行调用择优/串行融合/审查模式）、Prompt 模板库（版本管理+A/B测试）、多租户与团队管理（独立配额+审批流）

推荐从语义缓存开始（ROI最高，已有 embedding 基础设施铺垫）。

### 保留亮点
韧性五件套分层、ProviderAdapter 接口、RouteMatcher 可插拔、渐进式降级路由、TokenTracker 跨 chunk 解析、注释质量高、优雅关闭 + 健康检查分级。重构时不要破坏这些。
